# Домашнее задание к занятию «Микросервисы: подходы»

Вы работаете в крупной компании, которая строит систему на основе микросервисной архитектуры.
Вам как DevOps-специалисту необходимо выдвинуть предложение по организации инфраструктуры для разработки и эксплуатации.


## Задача 1: Обеспечить разработку

### Решение: **GitLab cloud (GitLab.com) + GitLab CI/CD + self-hosted GitLab Runner**

Облачная платформа, объединяющая систему контроля версий, непрерывную интеграцию и поставку. Решение позволяет централизованно управлять процессом разработки микросервисов и при этом выполнять сборки и тесты на собственной инфраструктуре компании.

#### Архитектура решения

- **GitLab (cloud)**  
  Используется для хранения Git-репозиториев (один репозиторий - один микросервис), управления merge request, пайплайнами CI/CD, шаблонами сборок и секретами.

- **GitLab CI/CD**  
  Отвечает за автоматический запуск сборок и тестов по событиям из системы контроля версий, а также за ручной запуск пайплайнов с параметрами. Поддерживает многостадийные и параллельные пайплайны.

- **GitLab Runner (self-hosted)**  
  Разворачивается на собственных серверах или в Kubernetes-кластере компании и выполняет сборки в Docker-контейнерах с использованием кастомных образов.


#### Соответствие требованиям

- Облачная система - используется GitLab SaaS  
- Система контроля версий Git  
- Отдельный репозиторий для каждого микросервиса  
- Запуск сборок по событиям (push, merge request, tag)  
- Ручной запуск сборок с указанием параметров  
- Индивидуальные настройки и переменные для каждой сборки  
- Поддержка шаблонов и переиспользуемых конфигураций пайплайнов  
- Безопасное хранение секретных данных  
- Несколько конфигураций сборки из одного репозитория  
- Возможность добавления кастомных шагов сборки  
- Использование собственных Docker-образов для сборки  
- Развёртывание агентов сборки на собственных серверах  
- Параллельный запуск нескольких сборок  
- Параллельный запуск тестов  

#### Обоснование выбора

GitLab предоставляет единую экосистему для DevOps-процессов, что снижает сложность сопровождения и интеграции инструментов. Платформа хорошо масштабируется под микросервисную архитектуру, позволяет стандартизировать CI/CD через шаблоны и обеспечивает безопасную работу с секретами. Использование self-hosted runner-ов даёт контроль над ресурсами и повышает безопасность.

#### Итог

GitLab SaaS в сочетании с GitLab CI/CD и self-hosted GitLab Runner является универсальным и масштабируемым решением, полностью удовлетворяющим требованиям к организации процесса разработки и эксплуатации микросервисов в крупной компании.


## Задача 2: Логи

### Решение: **Grafana Loki + Promtail + Grafana**

Лёгкое и облачно-ориентированное решение для централизованного сбора, хранения и анализа логов, хорошо подходящее для микросервисной архитектуры и контейнеризированных сред.


#### Архитектура решения

- **Promtail**  
  Агент сбора логов, устанавливаемый на все хосты (или как DaemonSet в Kubernetes). Считывает логи из stdout контейнеров и системных логов, добавляет метаданные и отправляет данные в Loki.

- **Grafana Loki**  
  Центральное хранилище логов. Принимает логи от Promtail, индексирует только метаданные, что снижает требования к ресурсам и упрощает масштабирование. Может быть развернут как в облаке, так и on-prem.

- **Grafana**  
  Пользовательский интерфейс для поиска, фильтрации и анализа логов. Используется для предоставления доступа разработчикам и эксплуатации.

#### Соответствие требованиям

- Централизованный сбор логов со всех хостов - Promtail собирает логи и отправляет их в единое хранилище Loki  
- Минимальные требования к приложениям - приложения пишут логи в stdout  
- Гарантированная доставка логов - буферизация и повторные попытки доставки в Promtail  
- Поиск и фильтрация логов - язык запросов LogQL  
- Пользовательский интерфейс для разработчиков — Grafana с разграничением прав доступа  
- Возможность делиться результатами поиска - сохранённые запросы и прямые ссылки в Grafana  

#### Обоснование выбора

- Loki не требует полного индексирования логов, что снижает нагрузку на хранилище и стоимость эксплуатации  
- Нативная работа с логами из stdout делает решение удобным для микросервисов и контейнеров  
- Grafana - фактически стандарт для observability с удобным UI и управлением доступами  
- Хорошо масштабируется и интегрируется с существующим стеком мониторинга (Prometheus, Alertmanager)  
- Меньшая операционная сложность по сравнению с классическим ELK-стеком

#### Итог

Связка Grafana Loki + Promtail + Grafana обеспечивает централизованный, надёжный и удобный сбор логов с минимальными требованиями к приложениям. Решение полностью соответствует требованиям микросервисной архитектуры и подходит для использования в крупной компании.

Решение должно соответствовать следующим требованиям:
- сбор метрик со всех хостов, обслуживающих систему;
