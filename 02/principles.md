
# Домашнее задание к занятию «Микросервисы: принципы»

Вы работаете в крупной компании, которая строит систему на основе микросервисной архитектуры.
Вам как DevOps-специалисту необходимо выдвинуть предложение по организации инфраструктуры для разработки и эксплуатации.

## Задача 1: API Gateway 

>Предложите решение для обеспечения реализации API Gateway. Составьте сравнительную таблицу возможностей различных программных решений. На основе таблицы сделайте выбор решения.
>
>Решение должно соответствовать следующим требованиям:
>- маршрутизация запросов к нужному сервису на основе конфигурации,
>- возможность проверки аутентификационной информации в запросах,
>- обеспечение терминации HTTPS.
>
>Обоснуйте свой выбор.

### Возможные решения
Рассмотрены следующие варианты:
- Kong
- NGINX (Open Source / Plus)
- Istio Ingress Gateway
- AWS API Gateway
- Traefik



### Сравнительная таблица

| Решение        | Маршрутизация | Аутентификация | HTTPS | Простота внедрения | Примечание |
|----------------|---------------|----------------|-------|--------------------|------------|
| **Kong**       | Да            | Да (плагины)   | Да    | Средняя            | API-first, плагины |
| **NGINX OSS**  | Да            | Ограниченно    | Да    | Высокая            | Мало security из коробки |
| **NGINX Plus** | Да            | Да             | Да    | Средняя            | Платный |
| **Istio**      | Да            | Да             | Да    | Низкая             | Сложен в настройке |
| **Traefik**    | Да            | Да             | Да    | Высокая            | Хорош для Kubernetes |


### Выбор решения: **Kong API Gateway**

#### Обоснование
- Полностью соответствует всем требованиям (маршрутизация, аутентификация, HTTPS).
- Поддерживает JWT, OAuth2 и другие механизмы через плагины.
- Хорошо интегрируется с Kubernetes и CI/CD.
- Подходит как для разработки, так и для промышленной эксплуатации.
- Имеет активное сообщество и подробную документацию.

**Итог:** Kong является оптимальным компромиссом между функциональностью, гибкостью и сложностью внедрения для микросервисной архитектуры.

## Задача 2: Брокер сообщений

>Составьте таблицу возможностей различных брокеров сообщений. На основе таблицы сделайте обоснованный выбор решения.
>
>Решение должно соответствовать следующим требованиям:
>- поддержка кластеризации для обеспечения надёжности,
>- хранение сообщений на диске в процессе доставки,
>- высокая скорость работы,
>- поддержка различных форматов сообщений,
>- разделение прав доступа к различным потокам сообщений,
>- простота эксплуатации.
>
>Обоснуйте свой выбор.

### Сравнение брокеров сообщений

| Критерий / Решение         | **Apache Kafka** | **RabbitMQ** | **Apache Pulsar** | **ActiveMQ** |
|----------------------------|------------------|--------------|------------------|---------------|
| **Кластеризация / HA**     | Да (репликация) | Да (кластер) | Да (BookKeeper)  | Да (кластеры) | 
| **Хранение на диске**      | Да, всегда       | Да (persistent) | Да, tiered storage | Да (опции) |
| **Высокая скорость**       | Очень высокая    | Средняя      | Высокая         | Низкая-средняя |
| **Форматы / протоколы**    | Kafka-протокол   | AMQP, MQTT, STOMP | Pulsar + gateway | AMQP, STOMP, MQTT |
| **Права доступа / безопасность** | ACL + RBAC | Плагины/ACL | Tenant + namespace ACL | ACL/JMS права |
| **Простота эксплуатации**  | Средняя-сложная | Простая-средняя | Средняя-сложная | Простая |

#### Краткие характеристики
- **Apache Kafka** - оптимизирован для **высокой скорости и надёжности**, хранит данные на диске, хорошо масштабируется через партиции и репликацию. Часто используется в event-ориентированных и stream-ориентированных системах.
- **RabbitMQ** - классический брокер с богатой поддержкой протоколов (AMQP, MQTT, STOMP), прост в настройке, хорошо подходит для типовых очередей задач.
- **Apache Pulsar** - современный брокер, разделяющий вычисления и хранение (BookKeeper), поддерживает multi-tenancy и высокую отказоустойчивость.
- **ActiveMQ** - зрелый брокер с поддержкой множества протоколов, но уступает остальным по производительности.


### Обоснование выбора

#### Рекомендованное решение: **Apache Kafka**

**Причины выбора:**
1. **Надёжность и кластеризация.** Kafka обеспечивает устойчивость за счёт репликации партиций и автоматического failover.
2. **Хранение сообщений на диске.** Сообщения пишутся в лог и могут храниться длительно (параметры retention).
3. **Производительность.** Kafka ориентирована на высокую пропускную способность и низкие задержки при больших объёмах.
4. **Права доступа и безопасность.** Поддерживаются ACL и RBAC, интеграции с Kerberos/SSL.
5. **Экосистема.** Широкий набор инструментов (Connect, Streams, Operators), помогает интегрировать брокер с DevOps-практиками.

**Альтернативы:**
- **RabbitMQ** - если важна простота эксплуатации и богатая поддержка протоколов (AMQP/MQTT).
- **Apache Pulsar** - если нужна мульти-тенантность, tiered storage и облачная масштабируемость.

**Вывод:** *Apache Kafka* оптимально совпадает с заданными требованиями по надёжности, скорости, хранению и управлению правами доступа, обеспечивая устойчивую работу микросервисной платформы.
